## Findings — gaps by owner (22 total)


### skills — the review's primary product  (15)

- (web) CometChat UIKit React v6 renders every component functional-but-UNSTYLED (no message bubbles, avatars/icons/call-buttons collapse to blank boxes, composer looks broken) when the app does not import the kit's design-token stylesheet '@cometchat/chat-uikit-react/css-variables.css'. The chat is fully operable (messages send/receive, calls place) so the failure is purely visual and silent.
- (web) Calls between clients fail immediately as 'Call Busy' / disconnect the instant they are initiated when EITHER side has a STALE/ghost active call in the SDK (a call surface that did not tear down, an app killed mid-call, or a lingering session). CometChat reports that user BUSY across ALL their sessions, so every new incoming call auto-rejects as 'Call Busy'. Neither the web nor the mobile UIKit clears a stale active call on (re)login, so a user can get permanently stuck 'busy' until the SDK state is reset. Aggravated by the same user being logged into multiple clients — any one busy/ghost session makes the user busy everywhere.
- (android) Kotlin-Views message-header call-button visibility API is misdocumented: skills/gotcha say hideVoiceCallButton/hideVideoCallButton (Compose-only); the Views CometChatMessageHeader has no such setters and requires setVoiceCallButtonVisibility(int)/setVideoCallButtonVisibility(int). Copying the documented names verbatim would not compile.
- (android) calls skill W3 tells integrators to stub 4 legacy com.cometchat.calls.* classes including RTCUser, but RTCUser ships in calls-sdk-android 5.0.1, so a 4th stub triggers a dex Duplicate class failure; only 3 stubs are correct.
- (android) Contradictory Calls-init guidance: builder-settings claims setEnableCalling(true) auto-inits the Calls SDK while gotcha X1 requires an explicit CometChatCalls.init, leaving integrators to guess. The agent worked around it by doing both.
- (android) Incoming-call overlay covers the conversation header (android). CometChatIncomingCall (UIKit v6.0.3) was embedded as an always-mounted match_parent overlay on the assumption it 'renders nothing until a call arrives'; it actually renders its Accept/Decline chrome unconditionally (no bound call), permanently covering the message header — peer identity, back button, and the voice/video call buttons — so a call can never be started from the thread. Verified with a freshly-provisioned user in a zero-history thread (0 incoming-call events in logcat) and with both parties of a 1:1 seeing it simultaneously. FIXED: keep the widget GONE and gate visibility on an incoming-call StateFlow<Call?> (set on onIncomingCallReceived, cleared on cancel/reject/accept/end) — the pattern the telehealth build used.
- (android) Placing a call crashes with 'Please call the CometChat.init() method ... preferably in the onCreate() method of the application class'. The app initializes the CometChat SDK lazily (ChatManager.ensureReady, on first chat open) instead of in Application.onCreate — MarketplaceApp.onCreate only calls ChatManager.install(), never CometChat/UIKit init. CometChat's calling UI runs in a SEPARATE PROCESS where the SDK was never initialized, so a UIKit view crashes: CometChatMessageComposer.onDetachedFromWindow -> CometChatMessageComposerViewModel.endTyping -> CometChat.getLoggedInUser -> SQLiteManager.getInstance throws (RuntimeException), taking down the call process. Confirmed: crash PID differed from the main app PID. FIX: cache the non-secret App ID + Region on first backend token fetch and call CometChatUIKit.init + CometChatCalls.init in Application.onCreate (runs in every process) so the SDK is initialized wherever a CometChat view mounts.
- (android) Android calling is impossible from the UI: the conversation screen renders edge-to-edge with the CometChat message header UNDER the status bar (no window insets applied). The header's voice/video call buttons sit in the top ~130px region where the system status-bar window consumes touches, so the buttons are untappable and a call can never be initiated. Confirmed live: tapping the voice button (bounds [891,45][954,108]) did nothing — no call initiated, an ONLINE 2nd client received no incoming call, no OutgoingCall event. After applying android:fitsSystemWindows=true to the ChatActivity root (header moves below the status bar, button now [891,181][954,244]), the SAME tap initiates the call: logcat 'Call Initiated Successfully', CometChatCallActivity (outgoing screen) mounts ringing the peer, and hang-up returns to the app. FIXED.
- (android) Group/conference calls stay stuck on 'connecting' (spinner never resolves) because CometChatCalls.joinSession runs before the calls CORE (CometChatCalls.init) is initialized in the current session, throwing 'Please call the CometChatCalls.init() method ...'. Root cause: (1) the eager per-process init bailed whenever the CHAT SDK was already initialized, never (re)initing the calls core; (2) the guard CometChatUIKit.isCallsSDKInitialized() reads true after UIKit setEnableCalling(true) even when CometChatCalls.init was never actually called this session, so the init was skipped. 1:1 calls work (they go through the chat SDK) which masks it — only the group/conference joinSession path exposes it.
- (android) 1:1 (and group) call UI breaks after minimise+reopen: the call renders correctly at first, but pressing home enters Picture-in-Picture (the UIKit declares CometChatCallActivity/CometChatOngoingCallActivity with supportsPictureInPicture=true), which resizes the React-Native/Jitsi call surface to the small PiP window; because those activities also declare configChanges=screenSize|screenLayout|orientation they handle the resize themselves and the RN surface never re-measures on return to full screen. Result: the whole call UI (participant tiles + controls) stays crammed in a narrow left strip with the rest of the screen black, while the call is actually still connected.
- (ios) [I7] iOS skill gotcha is wrong for CometChatSDK 4.1.6: it states CometChatStarscream is statically linked and an empty stub resolves the launch crash. In reality CometChatSDK.framework references a real Starscream symbol (WebSocketEvent.disconnected) at runtime; the empty stub only advances the crash from 'Library not loaded' to 'Symbol not found'. The real fix (pull CometChatStarscream 1.0.2 xcframework from CometChat CDN) is undocumented in skills.
- (ios) Calls skill is stale versus installed CometChatUIKitSwift 5.1.16 / CometChatSDK: it documents CallAppSettingsBuilder and generateToken(authToken:) which are deprecated/removed, forcing the agent to discover the current set(appID:)/set(region:)/generateToken(sessionID:) API against the .swiftinterface.
- (ios) iOS chat stuck on 'Connecting to chat…' after a cold login: ChatService.connect ran inside SwiftUI .task blocks, and the CometChat SDK init/login use withCheckedThrowingContinuation which does NOT observe Swift Task cancellation. When the hosting view updated (cancelling its .task) while suspended in a continuation, the continuation was orphaned — the await never returned, phase stuck at .connecting forever, POST /cometchat/token logged as cancelled.
- (ios) iOS CometChatMessageList renders skeleton loaders forever when a 1:1/group conversation is opened after a cold login: login succeeds (phase .ready), getUser/getGroup resolve (header+composer render), and the conversation LIST loads — but the message-HISTORY fetch never completes, so the thread shows shimmer placeholders indefinitely. The same accounts/conversations load fine on web. Re-setting the subject, adding a ConnectionListener re-fetch, and calling messageList.reload() do NOT recover it.
- (ios) iOS CometChatMessageList shows skeleton loaders forever (no timeout/error) when the conversation SUBJECT is set AFTER the list is already on-screen. The list fires its history fetch once, when it enters the window, ONLY if a subject (user/group) is already set. A common integration — create the list, add it to the view, then set the subject in an async getUser/getGroup callback — misses that window and never fetches, so the thread hangs on skeletons even though the socket is connected (calls deliver fine). Setting the subject synchronously from the known id before the view appears fixes it.

Additionally, the running build's dispute-GROUP conversation header has NO call buttons (group voice/video calling absent on web) though 1:1 does.
- (web) Gate 6 (cross-platform interop): a 1:1 VOICE call android-buyer -> web-seller does NOT connect. Signaling works — the caller's outgoing surface mounts, the web callee receives the incoming-call popup (.cometchat-incoming-call with __button-accept/__button-decline) and answering it (the REAL accept button, verified both via the Gate 6 harness AND a manual click) — but the media session never establishes: no ongoing-call UI mounts on either side, the android caller returns to chat, and the web call is logged as a 'Missed Call'. Reproduced 2x via gate6/run.py (choreography marketplace-1to1-voice fails at step 4 'call_connected', 30s timeout on BOTH clients) + once manually. The web client repeatedly logs the CometChat console error 'uiKitSettings not available' during call init — a concrete signal the web calling UIKit settings are not fully initialized (consistent with W5: the web CometChat chat/calling UI is missing from committed source; the running docker build is from an uncommitted/lost version). CAVEAT: a headless-Chromium <-> android-emulator WebRTC media path over docker/loopback could also contribute; the uiKitSettings signal + W5 point at incomplete web integration as the primary suspect.
- (android) auto_facts.build_pass=false with retry_count=3, yet the INTEGRATION SUMMARY claims a passing build gate. The delivered diff does not compile/build as-is; the overclaim of a green build gate is itself a defect and the root cause of the failure was not surfaced.
- (android) Placing or accepting a call intermittently 'redirects to the chats page': the app's own A3/A4 workaround (re-foreground MainActivity when a call ends) ran on ANY 'call ended' event/message with no session check, so a stale end message from an OLD unanswered call (e.g. the other side reloading their tab cancels its ring) arriving right as a NEW call starts brought the app task (top = Conversations) in front of the just-opened call screen. Compounding: the kit call activity's onUserLeaveHint then pushed the live call into PiP, and on return its RN/Jitsi surface stayed at PiP size (narrow-strip UI, A10).
- (ios) Build does not pass in the target environment (auto build_pass=false) after 7 retries, despite the INTEGRATION SUMMARY asserting a 'clean build and launch smoke'. Final integration state does not compile/link, and verification was over-reported.
- (ios) iOS dispute-GROUP chat is unreachable: tapping 'Open group chat & call' in ReportDetailView pops back to the Disputes queue instead of pushing the group ChatScreen — so group chat + group voice/video calling can never be opened on iOS. Reproduced 4x live (support login -> Disputes -> open a FLAGGED report whose inquiry is escalated -> scroll to the 'Dispute group' section -> tap 'Open group chat & call' -> lands back on the Disputes queue, two levels up). The 1:1 'Conversation' NavigationLink (InquiryDetailView -> ChatScreen(target:.user)) pushes correctly, so maestro/XCUITest CAN follow this app's NavigationLinks — the group link (ChatScreen(target:.group(guid:'dispute-<id>'))) specifically fails. Likely a SwiftUI navigation issue (the conditionally-rendered 'if inquiry.flagged' section's NavigationLink auto-popping on a data refresh, or the group ChatScreen throwing on connect). Root cause not yet isolated; behavior (group unreachable) is confirmed.

## Prioritized improvement suggestions

- **[skills]** (web) The integration agent must COMMIT the web chat UI it builds (the CometChat React components + /messages route). Verify Phase B completeness by rebuilding each component from committed source and re-running Gate 3 — a runtime green against a stale docker build hides an uncommitted integration. Also wire GROUP call buttons on the web message header (group voice/video), not just 1:1.
- **[skills]** (web) Re-integrate + COMMIT the web CometChat chat/calling UI (see W5) and ensure the web UIKit settings are initialized for calling (the 'uiKitSettings not available' console error must be gone). Then re-run Gate 6 (gate6/run.py one_to_one_voice) to confirm android<->web 1:1 voice actually connects. If it still fails with a committed, correctly-configured web build, escalate to a CometChat calls-SDK / cross-platform WebRTC interop investigation.
- **[skills]** (web) The web UIKit integration skill/quickstart MUST state that '@cometchat/chat-uikit-react/css-variables.css' has to be imported once app-wide (before local overrides); without it the entire kit UI is unstyled though functional. Add it to the minimal 'render your first component' snippet, not just a theming appendix.
- **[skills]** (web) The calling skill/quickstart should show clearing a stale active call on connect (CometChat.clearActiveCall() after login) and warn that getActiveCall() persisting from a prior/ungracefully-ended session makes the user permanently BUSY (every call auto-rejects 'Call Busy'). Also document multi-session busy semantics (a user is busy if ANY session has an active call) and that call-end/reject events must be reconciled against getActiveCall().sessionID before tearing down UI.
- **[skills]** (android) In the message-header skill (cometchat-android-v6-kotlin-components) and the /ui-kit/android/v6/message-header doc, split the call-button visibility API by kit: document setVoiceCallButtonVisibility(int)/setVideoCallButtonVisibility(int) for Kotlin Views and reserve hideVoiceCallButton/hideVideoCallButton for Compose only.
- **[skills]** (android) Correct the calls skill W3 stub list to exactly 3 classes (CometChatRTCView, RTCReceiver, RTCCallback) and explicitly warn NOT to stub RTCUser because it ships in calls-sdk-android 5.0.x and a stub collides at dex time.
- **[skills]** (android) Reconcile the builder-settings skill and gotcha X1 with a single authoritative statement on whether UIKitSettings.setEnableCalling(true) initializes the Calls SDK or an explicit CometChatCalls.init is required for the Views kit.
- **[docs-mcp]** (android) Add a docs-mcp page for the Go/gin backend auth-token flow (server-side createUser + mint auth token with the REST API key, returning appId/region/uid/authToken) so the expected backend auth is documentation-backed rather than left to sibling runs.
- **[skills]** (android) android-v6-kotlin skill/docs for CometChatIncomingCall must state the widget does NOT self-gate: show it only while a call is ringing (drive visibility from a call-state flow); never leave it mounted as a persistent overlay or it covers the header.
- **[skills]** (android) Skill/docs must warn that CometChat's calling UI runs in a separate process and the SDK MUST be initialized in Application.onCreate (every process), not lazily after login. When App ID/Region come from a backend, cache them (non-secret) and re-init synchronously on process start; only the auth-token login stays deferred.
- **[skills]** (android) android skill/docs must warn: when hosting the CometChat message header on a bare (no-Toolbar) screen, apply top system-window insets (fitsSystemWindows or WindowInsets padding) so the header — and its voice/video call buttons — is NOT under the status bar. Edge-to-edge is default on Android 15+, so without insets the call buttons land in the status-bar touch region and calling is impossible even though the buttons are visible.
- **[skills]** (android) Skill must make explicit that the Calls SDK core (CometChatCalls.init) is a SEPARATE, session/process-scoped init from both the Chat SDK and UIKit's setEnableCalling, and is REQUIRED before joinSession (group/conference calls). Guidance should NOT gate CometChatCalls.init on isCallsSDKInitialized()/isSDKInitialized() (those can read true without the calls core being live); CometChatCalls.init is idempotent, so re-assert it on every connect/app-start. Provide a group-call quickstart that shows this, since 1:1 calling hides the gap.
- **[skills]** (android) The UIKit's built-in call activities enable Picture-in-Picture but the React-Native/Jitsi call surface does not re-layout when leaving PiP (compounded by configChanges suppressing an activity recreate), leaving the call UI broken full-screen. Skill/kit should either (a) fix the PiP<->fullscreen re-measure of the call surface, or (b) document that apps must disable PiP on the call activities (manifest tools:replace supportsPictureInPicture=false) until fixed. Ship a working PiP sample or an explicit opt-out note. AMENDED 2026-07-14: the manifest opt-out (tools:replace supportsPictureInPicture=false) is NOT viable — the kit's onUserLeaveHint calls enterPictureInPictureMode() UNCONDITIONALLY, so with PiP disabled a home-press during a call throws IllegalStateException (CometChatCallActivity.kt:287) and CRASHES the app. Kit must (a) guard enterPictureInPictureMode (isPiP-allowed check / try-catch) and (b) re-measure the RN/Jitsi surface on PiP exit. Working app-side mitigation: keep PiP enabled + post a decorView.requestLayout() when the call activity resumes.
- **[skills]** (android) Workarounds that re-foreground/tear down on call-end signals MUST verify the ended sessionId matches CometChat.getActiveCall() and never yank focus while a call is live. The skill's call-lifecycle guidance should ship this guard pattern; end/cancel events for stale sessions are normal traffic, not exceptional.
- **[skills]** (ios) Rewrite the [I7] Starscream gotcha in the ios-core/ios-calls skill to version-gate: for CometChatSDK 4.1.x, CometChatStarscream is a real dynamic dependency and must be added as an explicit pod/xcframework from the CometChat CDN (include the library.cometchat.io xcframework URL); an empty stub does NOT fix the launch crash. This is the primary cause of the launch/build failure.
- **[skills]** (ios) Update the iOS calls skill to replace deprecated CallAppSettingsBuilder and generateToken(authToken:) with the current set(appID:)/set(region:) builder and generateToken(sessionID:), and note that the kit-default enable(inAppIncomingCall:true) path auto-renders incoming/ongoing UI (no ccShowOngoingCall).
- **[docs-mcp]** (ios) Fix the ios-uikit-quickstart docs bundle to stop referencing a composite CometChatMessages/MessagesVC and instead document composing CometChatMessageHeader + CometChatMessageList + CometChatMessageComposer, and add a Go/gin backend auth-token generation example so backend auth has doc coverage.
- **[skills]** (ios) The iOS ReportDetailView 'Open group chat & call' NavigationLink must reliably push the group ChatScreen. Move the destination out of the conditionally-rendered flagged section (or use a value-based navigationDestination / NavigationPath) so a data refresh from status changes doesn't auto-pop it, and verify the group ChatScreen connects (support IS a group member). Confirm group chat + group voice/video calling are reachable on iOS.
- **[skills]** (ios) Skill must warn that CometChat callback-based SDK calls wrapped in withCheckedThrowingContinuation are NOT cancellation-aware, so driving login/connect directly from a SwiftUI .task (which cancels on view updates) can orphan the continuation and hang the app on 'connecting'. Guidance: own the connect in a service-held Task (fire-and-forget, idempotent), not the view's task; or wrap continuations with withTaskCancellationHandler.
- **[skills]** (ios) The iOS UIKit message list can hang on skeleton loaders indefinitely (no timeout, no error surfaced) if its history fetch does not complete after a cold connect, while conversations + getUser succeed. Kit should (a) surface a fetch timeout/error state instead of infinite skeletons and (b) document a reliable re-fetch/recovery API — messageList.reload() did not re-arm the fetch. Needs a real-device repro to confirm vs simulator networking.
- **[skills]** (ios) The message-list skill must state that CometChatMessageList.set(user:)/set(group:) MUST be called BEFORE the view enters the window (e.g. synchronously in viewDidLoad from a User(uid:)/Group(guid:)) — setting the subject later, in an async user-fetch callback, misses the one-shot fetch trigger and the list stays on skeleton loaders with no error. Kit should also re-fetch when the subject is set post-appearance, or surface a timeout instead of infinite skeletons.
